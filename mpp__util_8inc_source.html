419  !> @brief Declare a pelist.
420  !!
421  !> This call is written specifically to accommodate a MPI restriction
422  !! that requires a parent communicator to create a child communicator, In
423  !! other words: a pelist cannot go off and declare a communicator, but
424  !! every PE in the parent, including those not in pelist(:), must get
425  !! together for the <TT>MPI_COMM_CREATE</TT> call. The parent is
426  !! typically <TT>MPI_COMM_WORLD</TT>, though it could also be a subset
427  !! that includes all PEs in <TT>pelist</TT>.
428  !!
429  !! This call implies synchronization across the PEs in the current
430  !! pelist, of which <TT>pelist</TT> is a subset.

432  integer, intent(in) :: pelist(:)
433  character(len=*), intent(in), optional :: name
434  integer :: i
435 
436  if( .NOT.module_is_initialized )call mpp_error( fatal, 'MPP_DECLARE_PELIST: You must first call mpp_init.' )
437  i = get_peset(pelist)
438  write( peset(i)%name,'(a,i2.2)' ) 'PElist', i !default name
439  if( PRESENT(name) )peset(i)%name = name
440  return

446  !!
447  !! This call sets the value of the current pelist, which is the
448  !! context for all subsequent "global" calls where the optional
449  !! <TT>pelist</TT> argument is omitted. All the PEs that are to be in the

459  !sync only across this subset. This is declared as the current pelist (peset(current_peset_num)%list)

508 !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
509  ! !
510  ! PERFORMANCE PROFILING CALLS !
511  ! !
512 !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
513 !predefined clock granularities, but you can use any integer!using CLOCK_LOOP and above may distort coarser-grain measurements
514  !> @brief Set the level of granularity of timing measurements.
515  !!
516  !> This routine and three other routines, mpp_clock_id, mpp_clock_begin(id),
517  !! and mpp_clock_end(id) may be used to time parallel code sections, and
518  !! extract parallel statistics. Clocks are identified by names, which
519  !! should be unique in the first 32 characters. The <TT>mpp_clock_id</TT>
520  !! call initializes a clock of a given name and returns an integer
521  !! <TT>id</TT>. This <TT>id</TT> can be used by subsequent
522  !! <TT>mpp_clock_begin</TT> and <TT>mpp_clock_end</TT> calls set around a
523  !! code section to be timed. Example:
524  !! <PRE>
525  !! integer :: id
526  !! id = mpp_clock_id( 'Atmosphere' )
527  !! call mpp_clock_begin(id)
528  !! call atmos_model()
529  !! call mpp_clock_end()
530  !! </PRE>
531  !! Two flags may be used to alter the behaviour of
532  !! <TT>mpp_clock</TT>. If the flag <TT>MPP_CLOCK_SYNC</TT> is turned on
533  !! by <TT>mpp_clock_id</TT>, the clock calls <TT>mpp_sync</TT> across all
534  !! the PEs in the current pelist at the top of the timed code section,
535  !! but allows each PE to complete the code section (and reach
536  !! <TT>mpp_clock_end</TT>) at different times. This allows us to measure
537  !! load imbalance for a given code section. Statistics are written to
538  !! <TT>stdout</TT> by <TT>mpp_exit</TT>.
539  !!
540  !! The flag <TT>MPP_CLOCK_DETAILED</TT> may be turned on by
541  !! <TT>mpp_clock_id</TT> to get detailed communication
542  !! profiles. Communication events of the types <TT>SEND, RECV, BROADCAST,
543  !! REDUCE</TT> and <TT>WAIT</TT> are separately measured for data volume
544  !! and time. Statistics are written to <TT>stdout</TT> by
545  !! <TT>mpp_exit</TT>, and individual PE info is also written to the file
546  !! <TT>mpp_clock.out.####</TT> where <TT>####</TT> is the PE id given by
547  !! <TT>mpp_pe</TT>.
548  !!
549  !! The flags <TT>MPP_CLOCK_SYNC</TT> and <TT>MPP_CLOCK_DETAILED</TT> are
550  !! integer parameters available by use association, and may be summed to
551  !! turn them both on.
552  !!
553  !! While the nesting of clocks is allowed, please note that turning on
554  !! the non-optional flags on inner clocks has certain subtle issues.
555  !! Turning on <TT>MPP_CLOCK_SYNC</TT> on an inner
556  !! clock may distort outer clock measurements of load imbalance. Turning
557  !! on <TT>MPP_CLOCK_DETAILED</TT> will stop detailed measurements on its
558  !! outer clock, since only one detailed clock may be active at one time.
559  !! Also, detailed clocks only time a certain number of events per clock
560  !! (currently 40000) to conserve memory. If this array overflows, a
561  !! warning message is printed, and subsequent events for this clock are
562  !! not timed.
563  !!
564  !! Timings are done using the <TT>f90</TT> standard
565  !! <TT>SYSTEM_CLOCK</TT> intrinsic.
566  !!
567  !! The resolution of SYSTEM_CLOCK is often too coarse for use except
568  !! across large swaths of code. On SGI systems this is transparently
569  !! overloaded with a higher resolution clock made available in a
570  !! non-portable fortran interface made available by
571  !! <TT>nsclock.c</TT>. This approach will eventually be extended to other
572  !! platforms.
573  !!
574  !! New behaviour added at the Havana release allows the user to embed
575  !! profiling calls at varying levels of granularity all over the code,
576  !! and for any particular run, set a threshold of granularity so that
577  !! finer-grained clocks become dormant.
578  !!
579  !! The threshold granularity is held in the private module variable
580  !! <TT>clock_grain</TT>. This value may be modified by the call
581  !! <TT>mpp_clock_set_grain</TT>, and affect clocks initiated by
582  !! subsequent calls to <TT>mpp_clock_id</TT>. The value of
583  !! <TT>clock_grain</TT> is set to an arbitrarily large number initially.
584  !!
585  !! Clocks initialized by <TT>mpp_clock_id</TT> can set a new optional
586  !! argument <TT>grain</TT> setting their granularity level. Clocks check
587  !! this level against the current value of <TT>clock_grain</TT>, and are
588  !! only triggered if they are <I>at or below ("coarser than")</I> the
589  !! threshold. Finer-grained clocks are dormant for that run.
590  !!
591  !!The following grain levels are pre-defined:
592  !!
593  !!<pre>
594  !!
595  !!
596  !! integer, parameter, public :: CLOCK_COMPONENT=1 !component level, e.g model, exchange
597  !! integer, parameter, public :: CLOCK_SUBCOMPONENT=11 !top level within a model component, e.g dynamics, physics
598  !! integer, parameter, public :: CLOCK_MODULE=21 !module level, e.g main subroutine of a physics module
599  !! integer, parameter, public :: CLOCK_ROUTINE=31 !level of individual subroutine or function
600  !! integer, parameter, public :: CLOCK_LOOP=41 !loops or blocks within a routine
601  !! integer, parameter, public :: CLOCK_INFRA=51 !infrastructure level, e.g halo update
602  !!</pre>
603  !!
604  !! Note that subsequent changes to <TT>clock_grain</TT> do not
605  !! change the status of already initiated clocks, and that if the
606  !! optional <TT>grain</TT> argument is absent, the clock is always
607  !! triggered. This guarantees backward compatibility.
608  subroutine mpp_clock_set_grain( grain )
609  integer, intent(in) :: grain
610  !set the granularity of times: only clocks whose grain is lower than
611  !clock_grain are triggered, finer-grained clocks are dormant.
612  !clock_grain is initialized to CLOCK_LOOP, so all clocks above the loop level
613  !are triggered if this is never called.
614  if( .NOT.module_is_initialized )call mpp_error( fatal, 'MPP_CLOCK_SET_GRAIN: You must first call mpp_init.' )
615 
616  clock_grain = grain
617  return
618  end subroutine mpp_clock_set_grain
