20 !> @defgroup mpp_domains_mod mpp_domains_mod
21 !> @ingroup mpp
22 !> @brief Domain decomposition and domain update for message-passing codes
23 !> @author V. Balaji SGI/GFDL Princeton University
24 !!
25 !> A set of simple calls for domain
26 !! decomposition and domain updates on rectilinear grids. It requires the
27 !! module mpp.F90, upon which it is built.\n
28 !! Scalable implementations of finite-difference codes are generally
29 !! based on decomposing the model domain into subdomains that are
30 !! distributed among processors. These domains will then be obliged to
31 !! exchange data at their boundaries if data dependencies are merely
32 !! nearest-neighbour, or may need to acquire information from the global
33 !! domain if there are extended data dependencies, as in the spectral
34 !! transform. The domain decomposition is a key operation in the
35 !! development of parallel codes.\n
36 !!\n
37 !! mpp_domains_mod provides a domain decomposition and domain
38 !! update API for rectilinear grids, built on top of the mpp_mod API for message passing.
39 !! Features of mpp_domains_mod include:\n
40 !!\n
41 !! Simple, minimal API, with free access to underlying API for more complicated stuff.\n
42 !!\n
43 !! Design toward typical use in climate/weather CFD codes.\n
44 !!
45 !> @par[Domains]
46 !! It is assumed that domain decomposition will mainly be in 2
47 !! horizontal dimensions, which will in general be the two
48 !! fastest-varying indices. There is a separate implementation of 1D
49 !! decomposition on the fastest-varying index, and 1D decomposition on
50 !! the second index, treated as a special case of 2D decomposition, is
51 !! also possible. We define domain as the grid associated with a <I>task</I>.
52 !! We define the compute domain as the set of gridpoints that are
53 !! computed by a task, and the data domain as the set of points
54 !! that are required by the task for the calculation. There can in
55 !! general be more than 1 task per PE, though often
56 !! the number of domains is the same as the processor count. We define
57 !! the global domain as the global computational domain of the
58 !! entire model (i.e, the same as the computational domain if run on a
59 !! single processor). 2D domains are defined using a derived type domain2D,
60 !! constructed as follows (see comments in code for more details).
61 !!
62 !! type, public :: domain_axis_spec
63 !! private
64 !! integer :: begin, end, size, max_size
65 !! logical :: is_global
66 !! end type domain_axis_spec
67 !!
68 !! type, public :: domain1D
69 !! private
70 !! type(domain_axis_spec) :: compute, data, global, active
71 !! logical :: mustputb, mustgetb, mustputf, mustgetf, folded
72 !! type(domain1D), pointer, dimension(:) :: list
73 !! integer :: pe ! pe to which the domain is assigned
74 !! integer :: pos
75 !! end type domain1D
76 !!
77 !! type, public :: domain2D
78 !! private
79 !! type(domain1D) :: x
80 !! type(domain1D) :: y
81 !! type(domain2D), pointer, dimension(:) :: list
82 !! integer :: pe ! PE to which this domain is assigned
83 !! integer :: pos
84 !! end type domain2D
85 !!
86 !! type(domain1D), public :: NULL_DOMAIN1D

774  !> @brief Set up a domain decomposition.
775  !!
776  !> There are two forms for the \e mpp_define_domains call. The 2D version is generally
777  !! to be used but is built by repeated calls to the 1D version, also provided.
778  !!
779  !! <br>Example usage:
780  !!
781  !! call mpp_define_domains( global_indices, ndivs, domain, &
782  !! pelist, flags, halo, extent, maskmap )
783  !! call mpp_define_domains( global_indices, layout, domain, pelist, &
784  !! xflags, yflags, xhalo, yhalo, &
785  !! xextent, yextent, maskmap, name )
786  !!
787  !! @param global_indices Defines the global domain.
788  !! @param ndivs The number of domain divisions required.
789  !! @param [inout] domain Holds the resulting domain decomposition.
790  !! @param pelist List of PEs to which the domains are to be assigned.
791  !! @param flags An optional flag to pass additional information
792  !! about the desired domain topology. Useful flags in a 1D decomposition
793  !! include <TT>GLOBAL_DATA_DOMAIN</TT> and
794  !! <TT>CYCLIC_GLOBAL_DOMAIN</TT>. Flags are integers: multiple flags may
795  !! be added together. The flag values are public parameters available by
796  !! use association.
797  !! @param halo Width of the halo.
798  !! @param extent Normally <TT>mpp_define_domains</TT> attempts
799  !! an even division of the global domain across <TT>ndivs</TT>
800  !! domains. The <TT>extent</TT> array can be used by the user to pass a
801  !! custom domain division. The <TT>extent</TT> array has <TT>ndivs</TT>
802  !! elements and holds the compute domain widths, which should add up to
803  !! cover the global domain exactly.
804  !! @param maskmap Some divisions may be masked
805  !! (<TT>maskmap=.FALSE.</TT>) to exclude them from the computation (e.g
806  !! for ocean model domains that are all land). The <TT>maskmap</TT> array
807  !! is dimensioned <TT>ndivs</TT> and contains <TT>.TRUE.</TT> values for
808  !! any domain that must be <I>included</I> in the computation (default
809  !! all). The <TT>pelist</TT> array length should match the number of
810  !! domains included in the computation.
811  !!
812  !! <br>Example usage:
813  !! @code{.F90}
814  !! call mpp_define_domains( (/1,100/), 10, domain, &
815  !! flags=GLOBAL_DATA_DOMAIN+CYCLIC_GLOBAL_DOMAIN, halo=2 )
816  !! @endcode
817  !!
818  !! defines 10 compute domains spanning the range [1,100] of the global
819  !! domain. The compute domains are non-overlapping blocks of 10. All the data
820  !! domains are global, and with a halo of 2 span the range [-1:102]. And
821  !! since the global domain has been declared to be cyclic,
822  !! domain(9)%next => domain(0) and domain(0)%prev =>
823  !! domain(9). A field is allocated on the data domain, and computations proceed on
824  !! the compute domain. A call to mpp_update_domains would fill in the values
825  !! in the halo region:<br>
826  !!<br>
827  !! @code{.F90}
828  !! call mpp_get_data_domain( domain, isd, ied ) !returns -1 and 102
829  !! call mpp_get_compute_domain( domain, is, ie ) !returns (1,10) on PE 0 ...
830  !! allocate( a(isd:ied) )
831  !! do i = is,ie
832  !! a(i) = &lt;perform computations&gt;
833  !! end do
834  !! call mpp_update_domains( a, domain )
835  !! @endcode
836  !!<br>
837  !! The call to mpp_update_domainsfills in the regions outside
838  !! the compute domain. Since the global domain is cyclic, the values at
839  !! \e i=(-1,0) are the same as at \e i=(99,100); and \e i=(101,102)
840  !! are the same as \e i=(1,2).
841  !!
842  !! The 2D version is just an extension of this syntax to two dimensions.
843  !!
844  !! The 2D version of the above should generally be used in
845  !! codes, including 1D-decomposed ones, if there is a possibility of
846  !! future evolution toward 2D decomposition. The arguments are similar to
847  !! the 1D case, except that now we have optional arguments
848  !! flags, halo, extent and maskmap along two axes.
849  !!
850  !! flags can now take an additional possible value to fold one or more edges.
851  !! This is done by using flags \e FOLD_WEST_EDGE, \e FOLD_EAST_EDGE, \e FOLD_SOUTH_EDGE or
852  !! \e FOLD_NORTH_EDGE. When a fold exists (e.g cylindrical domain),
853  !! vector fields reverse sign upon
854  !! crossing the fold. This parity reversal is performed only in the vector version of
855  !! mpp_update_domains. In addition, shift operations may need to be applied to vector fields on
856  !! staggered grids, also described in the vector interface to mpp_update_domains.
857  !!
858  !! <TT>name</TT> is the name associated with the decomposition,
859  !! e.g <TT>'Ocean model'</TT>. If this argument is present,
860  !! <TT>mpp_define_domains</TT> will print the domain decomposition
861  !! generated to <TT>stdlog</TT>.
862  !!
863  !! <br>Examples:
864  !! call mpp_define_domains( (/1,100,1,100/), (/2,2/), domain, xhalo=1 )
865  !! will create the following domain layout:<br>
866  !!<br>
867  !! | domain |domain(1)|domain(2) |domain(3) |domain(4) |
868  !! |--------------|---------|-----------|-----------|-------------|
869  !! |Compute domain|1,50,1,50|51,100,1,50|1,50,51,100|51,100,51,100|
870  !! |Data domain |0,51,1,50|50,101,1,50|0,51,51,100|50,101,51,100|
871  !!
872  !!
873  !! Again, we allocate arrays on the data domain, perform computations
874  !! on the compute domain, and call mpp_update_domains to update the halo region.
875  !!
876  !! If we wished to perfom a 1D decomposition along Y on the same global domain,
877  !! we could use:
878  !!
879  !! call mpp_define_domains( (/1,100,1,100/), layout=(/4,1/), domain, xhalo=1 )

882  !!<br>

886  !! |Data domain |0,101,1,25|0,101,26,50|0,101,51,75|1,101,76,100|

935 !> Performs halo updates for a given domain.<br>
936 !!
937 !! Used to perform a halo update of a
938 !! domain-decomposed array on each PE. \e MPP_TYPE can be of type
939 !! complex, integer, logical or real of 4-byte or 8-byte kind; of rank up to 5.
940 !! The vector version (with two input data fields) is only present for real types.
941 !! For 2D domain updates, if there are halos present along both
942 !! x and y, we can choose to update one only, by specifying \e flags=XUPDATE or \e flags=YUPDATE.
943 !! In addition, one-sided updates can be performed by setting flags
944 !! to any combination of WUPDATE, EUPDATE, SUPDATE and NUPDATE
945 !! to update the west, east, north and south halos respectively.
946 !! Any combination of halos may be used by adding the requisite flags, e.g:
947 !! \e flags=XUPDATE+SUPDATE or \e flags=EUPDATE+WUPDATE+SUPDATE will update the east,
948 !! west and south halos.<br>
949 !!<br>
950 !! If a call to \e mpp_update_domains involves at least one E-W
951 !! halo and one N-S halo, the corners involved will also be updated, i.e,
952 !! in the example above, the SE and SW corners will be updated.<br>
953 !! If \e flags is not supplied, that is equivalent to \e flags=XUPDATE+YUPDATE.<br>
954 !!<br>
955 !! The vector version is passed the \e x and \e y components of a vector field in tandem,
956 !! and both are updated upon return. They are passed together to treat parity issues on various
957 !! grids. For example, on a cubic sphere projection, the \e x \e y components may be
958 !! interchanged when passing from an equatorial cube face to a polar face.
959 !! For grids with folds, vector components change sign on crossing the fold. Paired scalar
960 !! quantities can also be passed with the vector version if \e flags=SCALAR_PAIR, in which
961 !! case components are appropriately interchanged, but signs are not.<br>
962 !!<br>
963 !! Special treatment at boundaries such as folds is also required for
964 !! staggered grids. The following types of staggered grids are
965 !! recognized:<br>
